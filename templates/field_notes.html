Preventing Display Sleep on an Unattended Dashboard

Business Need: The contact center displayed real-time call statistics on a large screen 24/7.

Challenge: Windows power settings alone were not sufficient to keep the display active indefinitely.

Solution: I created a lightweight PowerShell script that simulated a Shift key press every 30 seconds, keeping the system from ever entering an idle state.

Outcome: The dashboard remained visible throughout unattended shifts without requiring third-party software or manual intervention.

Sometimes the simplest automation provides the greatest operational value.

When the C: Drive Looks Empty but Is Actually Full

One of the more confusing support cases I encountered involved laptops that became painfully slow over time. Users reported a small Windows warning near the taskbar saying the C: drive was almost full — easy to dismiss, and many did, until the machine became too slow for normal work.

The Environment: Our corporate Windows images were intentionally configured to reduce the risk of accidental system changes. Users could technically reach C: by typing the path manually, but Explorer was set up so they primarily saw their own profile under C:\Users\<username>, which discouraged browsing into folders like Program Files, Windows, or ProgramData. Some generations of the image went further, keeping Windows on C: and user data on a dedicated D: partition.

The Investigation: One laptop arrived at 99% disk utilization. The obvious suspect — the user's Documents or Downloads — turned out to hold less than 1 GB; the employee had already deleted files trying to fix it themselves. Digging deeper, the real culprit was hidden inside C:\ProgramData: years of accumulated antivirus scan logs. Normally the security software cleaned these up once the machine reconnected to the internal network, but laptops that stayed off-site for long stretches just kept collecting logs until the drive was nearly full.

Why It Was Hard to Diagnose: Users couldn't easily see hidden system folders, their personal files barely took up any space, remote support tools gave limited visibility over the public Internet, and the "low disk space" warning never said where the space had actually gone. The visible symptom and the real cause pointed in completely different directions.

If Windows reports the system drive is full but the user's profile barely holds any data, don't assume the user is responsible. Hidden system folders like ProgramData can quietly consume tens of gigabytes without anyone noticing.

When Only the Company Laptop Won't Connect to Wi-Fi

One support case taught me an important lesson: just because a problem looks like it's outside your support boundary doesn't mean the root cause is.

The Situation: As remote work became more common, our company issued laptops that connected securely to corporate resources over VPN — but only after meeting strict conditions: up-to-date antivirus definitions, current intrusion detection/prevention signatures, required Windows updates, and corporate security policy compliance. Then a new kind of ticket started showing up: "My company laptop won't connect to my home Wi-Fi." Every other device in the house — personal laptops, phones, tablets, smart TVs — connected fine. Only the company laptop failed.

Looking Beyond the Obvious: It would have been easy to say "contact your ISP" and close the ticket. But after asking a few more questions, a pattern emerged: almost every affected user had recently upgraded their home Internet service or replaced their router. The timing wasn't a coincidence — many providers had started shipping Wi-Fi 6 and Wi-Fi 6E routers, while the company laptops were still running older wireless drivers that didn't fully support the newer standards.

The Real Cause: The hardware itself was compatible; what was missing was the wireless driver. Because standard users weren't permitted to install device drivers, the update needed administrator privileges. Once the driver was updated, the laptops connected to the new routers with no other changes needed. The issue was never the user's Internet service — it was the company's managed endpoint.

Good technical support isn't about proving where responsibility ends. It's about asking enough questions to discover where the real problem begins.

Network Outages — Root Causes
🌐
Automated Network Health Monitoring with PowerShell

Before this automation existed, checking network device health meant manually pinging each device one by one while cross-referencing a network diagram — slow, repetitive, and easy to make mistakes on (skipped devices, missed timeouts) during a long shift.

I built a PowerShell-based monitoring script that pings every registered device (-n 10 -w 4000) in sequence, then loops continuously. If 5 consecutive pings time out, the script flags it as a network incident, writes a timestamped entry to a log file, and turns the terminal background red as a visual alert. 

function Test-TimedOut {
    param([string[]]$pingOutput)
    $timedOutCount = ($pingOutput | Select-String `
        -Pattern "Timed Out|Request timed out").Count
    return $timedOutCount
}

if ($timedOutCount -ge $failThreshold) {
    Write-IncidentLog -target $target
    $Host.UI.RawUI.BackgroundColor = "Red"
}

The log file is overwritten on each incident rather than appended, so checking it the next morning immediately shows the timestamp of the most recent outage — no need to scroll through a growing log to figure out whether an issue is still ongoing.

I also built per-site launcher tools so a single click could sweep every device at a site, or target just one — for example, monitoring only the LAN-side interface of a backup modem.

Even though network infrastructure was formally owned by an external network provider, this tool let our team detect — and confirm recovery from — outages just as fast, and sometimes faster. It also cut down on human error from the old manual-ping workflow, like accidentally skipping a device during a check.

Isolating a Client-Side Root Cause Behind a Suspected Server Outage

Situation: We started receiving reports from field locations across North America that a mission-critical business web application was unavailable. Because the reports came from multiple locations at once, it initially looked like a server-side outage.

Task: My responsibility was to identify the root cause as quickly as possible, since the application was operationally critical.

Action: Instead of assuming the server was down, I noticed that a small number of PCs were still functioning normally — which suggested the problem might be client-side rather than server-side. I built a test PC from an older system image and confirmed the application worked correctly on it. I then installed recent security patches one at a time until the issue reappeared, which isolated the problem to a specific NetBIOS over TCP/IP security patch. Reviewing the patch documentation confirmed it was intended for servers but had been mistakenly deployed to client PCs.

Result: I shared the findings with our engineering team, who halted further deployment, instructed users not to reboot unaffected PCs, and distributed a corrected package. Because the root cause was identified quickly, the broader support organization was able to recover the service much faster than an unexplained outage would normally allow. I later received recognition from the engineering team for the investigation.

A symptom reported from many locations at once doesn't automatically mean the root cause is server-side — ruling out the client is often the faster path to the real answer.

Network Infrastructure & Hidden Devices
🔌
The Hidden Risk of Unmanaged Switches and Hubs

One of the most overlooked causes of network performance problems is the small unmanaged switch or hub hidden under a desk.

Several years ago, users began reporting that network connections felt unusually slow. Basic troubleshooting showed no obvious issues. Switch ports looked healthy, servers were operating normally, and there were no obvious network errors.

The root cause turned out to be something completely unexpected.

Someone had installed an old 10 Mbps or 100 Mbps unmanaged switch years earlier to solve a simple cabling problem. Furniture had been rearranged, adding another network drop would have required construction work, and a small inexpensive switch was the quickest solution. Unfortunately, no one documented it, and over time the device became forgotten.

For light office traffic such as web browsing or email, the limitation wasn't obvious. However, when multiple users began downloading large files or transferring data simultaneously, the old switch became a significant bottleneck. Even worse, because unmanaged switches cannot be monitored through enterprise network management systems, they often remain invisible until users begin reporting performance problems.

Why They Become a Problem

Small unmanaged switches create several operational challenges:

  • They cannot be monitored remotely.
  • They provide little or no diagnostic information.
  • Older models may only support 10/100 Mbps speeds.
  • They frequently fail after years of continuous operation.
  • They are often hidden beneath desks or inside furniture, making troubleshooting difficult.
  • Spare network ports may introduce unnecessary security risks if unauthorized devices are connected.

During office relocations, another common issue appears. Initial planning may underestimate the number of required network ports, leading staff to install inexpensive unmanaged switches as temporary solutions. Unfortunately, these temporary fixes often become permanent.

Lessons Learned

Whenever office layouts were updated or equipment was relocated, I made it a priority to identify and remove unmanaged switches whenever possible. Replacing them with properly managed network connections improved visibility, simplified troubleshooting, reduced security risks, and eliminated hidden bandwidth bottlenecks.

The smallest device under a desk can sometimes become the biggest obstacle to network reliability.

If users report intermittent "slow network" issues but monitoring tools show everything is healthy, don't forget to look under the desks. Sometimes the problem isn't in the data center — it is hiding beneath someone's workstation.

Security & Access Control
🔐
PowerShell Execution Policy Handling

Challenge: While developing VirtualHelpDesk, I needed to launch PowerShell automation scripts from a batch file. Windows initially blocked the script with an error indicating that running scripts was disabled on the system. This happens because the default PowerShell Execution Policy prevents .ps1 files from running.

Investigation: I checked the current policy with Get-ExecutionPolicy, which confirmed the system was configured to restrict script execution — a common Windows default.

Initial Solution: For development purposes, I temporarily enabled script execution for my user profile, which resolved the issue and let the batch file launch the script.

Improving the Design: That approach wasn't suitable for automation, since changing the execution policy can require user confirmation and isn't ideal for enterprise environments. I redesigned the launcher to use a process-scoped policy override instead: 

powershell.exe -ExecutionPolicy Bypass -File Start.ps1

This approach applies only to the current PowerShell process, doesn't permanently modify the user's execution policy, and lets the automation run without interactive prompts — better for demonstrations and automated workflows.

Enterprise Considerations: In a production environment, I wouldn't rely on permanently changing execution policies on individual machines. Organizations typically manage this through Group Policy, code-signing certificates for trusted scripts, or process-level overrides scoped to specific deployment tools.

This helped me understand the balance between automation, security, and usability — and why process-level execution is often preferred for portable automation projects like VirtualHelpDesk.
